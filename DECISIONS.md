# DECISIONS.md — Non-Obvious Architectural Decisions

One line + reason each. Keeps the system explainable (PROJECT_BRIEF.md section 6, item 5).

- **`StructuredError` is a discriminated union keyed by `type`, not a flat interface with `details: object`.** Reason: SDK_API.md section 5 requires full type safety for TypeScript consumers; a union lets `details` narrow correctly (e.g. `error.details.package` only available when `error.type === "missing_dependency"`) without any casting.

- **`RawResult.durationMs` is optional; `analyze()` will default it to `0` when absent.** Reason: SDK_API.md section 3's `analyze()` example passes no duration (the caller only has `command`/`exitCode`/`stdout`/`stderr`), but `CaptureResult.durationMs` is a required field per ERROR_SCHEMA.md section 2. Making it optional on the input and defaulting on the output resolves the gap without changing either contract file. `capture()` will always supply the real measured value.

- **Two tsconfig files: `tsconfig.json` (noEmit, includes `src` + `tests`, used for typecheck/editor) and `tsconfig.build.json` (extends it, emits from `src` only into `dist`).** Reason: keeps test files type-checked without polluting the published `dist/` output.

- **Test runner: Vitest.** Reason: not specified in the three contract files; PROJECT_BRIEF.md section 4.6 requires at least one test per feature. Vitest was chosen for native ESM/TS support with no separate transpile step, keeping the "boring core" simple.

- **Module format: ESM (`"type": "module"`, `NodeNext` resolution).** Reason: not specified in the contract files; chosen as the modern default for a new Node.js library. Revisit only if a real consumer needs CJS.

- **Dependency-cruiser boundaries enforce one-directional imports: `types` ← `classifiers`/`core`/`sdk`; `classifiers` → `types` only (never each other, never `core`/`sdk`); `core` → `types` + `classifiers` (never `sdk`); `sdk` → `core` + `types` (never individual classifiers).** Reason: directly encodes PROJECT_BRIEF.md section 4.1 ("modules talk only through interfaces") and SDK_API.md section 6 ("classifiers never reach into the core or into each other").

- **`AnalyzeOptions` and `UnknownErrorDetails` are `Record<string, never>` rather than an empty `interface {}` as literally shown in SDK_API.md section 4 / ERROR_SCHEMA.md section 4.8.** Reason: an empty TS interface accepts almost any object structurally (it imposes no constraint), and ESLint's `no-empty-object-type` rule (part of `stylisticTypeChecked`) flags `interface Foo {}` outright. `Record<string, never>` means the same thing — "no properties" — while actually being enforced by the type system.

- **The `Classifier` interface lives in `/src/types`, not `/src/core`.** Reason: `SDK_API.md` section 6 calls it "internal, but the key extension point," and the dependency-cruiser boundary forbids `classifiers → core`. Since classifier files must implement `Classifier`, the interface has to live somewhere classifiers are allowed to import from — `/types` is that leaf module.

- **The registry's `unknown_error` fallback treats `exitCode !== 0` as failure (covering both non-zero and `null`).** Reason: matches `ERROR_SCHEMA.md` section 2's own definition of `success` (`exitCode === 0`, else `false`), and section 4.8's rule that every failed command must produce at least one `StructuredError`. A `null` exit code (process killed/never exited) is still a failure, so it also gets the fallback.

- **`runClassifiers` returns `[]` unconditionally on a successful command, discarding anything a classifier produced.** Reason: `ERROR_SCHEMA.md` section 2 requires `errors` to be empty whenever `success` is true. Rather than trusting every future classifier author to only match on failing output, the registry enforces this invariant itself — the one place all classifier output passes through. Added after code review flagged that a misbehaving classifier could otherwise leak errors into a successful `CaptureResult`.

- **`CaptureOptions` and `AnalyzeOptions` are exported from the public surface even though SDK_API.md section 5's literal export list only names `CaptureResult, StructuredError, RawResult, ErrorType`.** Reason: section 4 defines both types as part of `capture()`/`analyze()`'s own public signatures, and section 1's design goal is "predictable return... full type safety." Omitting them would leave consumers unable to name the options type when building one up (e.g. `const opts: CaptureOptions = {...}`), which contradicts the "full type safety" goal in section 5's own header. Read as an oversight in section 5's list rather than a deliberate exclusion; `Classifier` stays internal since section 6 explicitly labels it "internal."

- **`UsageError` is thrown by `capture()` but not exported from the public surface.** Reason: SDK_API.md section 5's export list is explicit and doesn't include it, and section 1's design goal is a tiny surface. Consumers can still distinguish it from a normal rejection via `error.name === "UsageError"` (or `error instanceof Error`, since it extends `Error`). Revisit if real usage shows consumers need `instanceof` checks.

- **`capture()` uses `node:child_process`'s `exec` (shell-mode), not `spawn`/`execFile`.** Reason: SDK_API.md's own examples pass full shell command strings (`capture("npm install")`, `capture("npm run build")`), which need shell interpretation (PATH resolution, `&&`, etc.) that `execFile`/`spawn` without `shell: true` don't provide. `exec`'s callback signature also makes it natural to capture stdout/stderr/exit code unconditionally without a stray rejection on non-zero exit — `capture()` builds the `RawResult` from the callback's `(error, stdout, stderr)` regardless of whether `error` is set, treating a non-numeric `error.code` (e.g. killed by `timeoutMs`) as `exitCode: null`, consistent with ERROR_SCHEMA.md section 2's `exitCode: number | null`.

- **`capture()` sets `exec()`'s `maxBuffer` to 20MB instead of Node's 1MB default.** Reason: code review found that the default silently truncates stdout/stderr and reports `exitCode: null` — identical in shape to a genuine `timeoutMs` kill, and a direct violation of `ERROR_SCHEMA.md` section 5's "`raw` is never dropped" rule. `npm install`/build/test output routinely exceeds 1MB with verbose or CI logging (exactly `PROJECT_BRIEF.md`'s primary use case), so the cap is raised well above realistic output size. Not exposed as a `CaptureOptions` field — it's an internal implementation default, not a documented option in SDK_API.md section 4.

- **`src/index.ts` (package entry point) now exists, added in Phase 4.** Reason: it needs `capture`/`analyze`, which didn't exist until this phase (the earlier DECISIONS.md entry deferring it is superseded by this one). It's a one-line re-export of `src/sdk/index.ts`, matching `package.json`'s existing `main`/`types` fields (`./dist/index.js`).

- **The four tsc-diagnostic classifiers (`missing_dependency`, `module_not_found`, `type_error`, `syntax_error`) each match TWO diagnostic formats, not just the one in ERROR_SCHEMA.md's example.** Reason: found by an end-to-end smoke test against *real* `tsc` output (not a hand-built fixture) run through the built `dist/index.js`. `tsc` only emits the "pretty" `file:line:col - error TSxxxx: ...` format (the one in ERROR_SCHEMA.md section 6) when `--pretty` is explicitly passed or it's attached to a real TTY. Since `capture()` always runs commands through a piped, non-interactive `child_process.exec` — never a TTY — real `tsc` invocations actually emit `file(line,col): error TSxxxx: ...` by default. Without handling both, every TS-diagnostic classifier would have silently never fired in real usage despite passing on the schema's own example and hand-built test fixtures. All four classifiers now try both formats; regression tests added for each using the plain format.

- **`missing_dependency`/`module_not_found` duplicate their TS-diagnostic and Node-require-error parsing logic instead of sharing one function.** Reason: real tension between PROJECT_BRIEF.md section 4.1 ("no duplicated logic — appears twice, extract a shared function") and the dependency-cruiser boundary / SDK_API.md section 6 ("classifiers never reach into... each other"). A shared helper can't live in `/classifiers` (any file there other than `index.ts` is blocked from being imported by a sibling), and putting parsing logic in `/types` would blur that folder's purpose (schema/contracts, not behavior). Resolved in favor of the stronger, more explicit architectural constraint (classifier isolation) over strict DRY for what is a small, non-drifting amount of duplication (a regex + a 3-line specifier check). Flagging this explicitly rather than silently picking a side, since it's a genuine trade-off between two brief rules.

- **`install_failure` whitelists specific npm error codes (`ERESOLVE`, `ENOTFOUND`, `E404`, etc.) rather than matching any `npm ERR!` line.** Reason: `npm` prints `npm ERR! code ELIFECYCLE` whenever *any* lifecycle script fails, including in ERROR_SCHEMA.md section 6's own north-star example (a `tsc` failure inside `npm run build`) — which must classify as `missing_dependency` alone. Matching on any `npm ERR!` would have made `install_failure` fire on that example too, producing two errors instead of one. A regression test locks in that `install_failure` does NOT match the section 6 example's stderr.

- **`port_in_use`'s `suggestedFix` avoids naming a specific kill command (e.g. `lsof -ti:PORT | xargs kill`).** Reason: `suggestedFix` text could plausibly be executed by an agent unattended; a force-kill command is destructive, platform-specific (POSIX only), and not something this SDK should hand out as "the" fix. Uses a conservative textual suggestion instead.

- **`command_not_found`'s bash pattern also accepts a `line N:` prefix** (`build.sh: line 3: vite: command not found`), **not just a bare digit prefix** (`sh: 1: vite: not found`). Reason: code review found the `line N:` phrasing — bash's actual wording when a script is run non-interactively (CI, Docker, any npm script that shells out to a `.sh` file) — wasn't matched at all, silently falling through to `unknown_error`. This is more common in practice than the bare-digit `sh:` form the classifier already handled. Confirmed fixed against a real subprocess through the built `dist/index.js`.

- **`missing_dependency`/`module_not_found`'s Node-require-error regex captures the "Require stack:" file in the SAME match as its error, instead of a separate global scan for "the" stack block.** Reason: code review found that with two distinct `require()` failures in one capture, the old separate-scan approach always paired both errors with the FIRST stack block, mislabeling the second error's file. Fixed by making the require-stack group part of one combined, optional capture group on the same regex — each match is now self-contained. Low real-world likelihood for a single Node process (which normally crashes on the first uncaught require failure), but cheap and correct to fix regardless; regression tests added for both files.

- **`install_failure` pairs each `npm ERR! code X` line with its own immediately-following `npm ERR! <message>` line (one combined regex), and picks the LAST whitelisted block when several appear, instead of always the first.** Reason: code review found two related bugs from the original separate-scan approach: (1) with two `npm ERR! code` blocks, "the" message was always grabbed from the first block regardless of which code it paired with — the same class of bug as the require-stack one above; (2) an early transient whitelisted code (e.g. a retried `ETIMEDOUT`) would shadow the real terminal failure that follows it. A code with no adjacent message line still falls back to the bare code line, preserving the original fallback behavior.

- **License: MIT.** Reason: not specified anywhere in the three contract files. MIT is the overwhelmingly common default for a small, permissive npm library like this one; low-risk, standard choice. Added `LICENSE` and `package.json`'s `license` field. Revisit if the user has a different preference.

- **`package.json` omits `repository`/`homepage`/`bugs`.** Reason: no real repository URL exists yet (this project isn't pushed anywhere) — inventing one would be a fabricated link. Add once a real remote exists.

- **`package.json`'s `files` field includes `ERROR_SCHEMA.md`, `ARCHITECTURE.md`, `DECISIONS.md`, and `docs/` alongside `dist/`.** Reason: README links to all of these directly, and without a `repository` field (see above) npm can't rewrite relative markdown links to point at a GitHub source — code review caught that those links would 404 for anyone reading the published README on npmjs.com or a locally installed copy. Shipping the files themselves means the links resolve for a local install (`node_modules/error-capture-sdk/ARCHITECTURE.md` etc. actually exists) even though they still won't resolve on the npmjs.com *rendered page* itself (npm doesn't serve arbitrary tarball contents as clickable links there — only a real `repository` field fixes that). Verified via `npm pack`: the published tarball now contains `dist/` + `README.md` + `LICENSE` + `ERROR_SCHEMA.md` + `ARCHITECTURE.md` + `DECISIONS.md` + `docs/CLASSIFIER_GUIDE.md` + `package.json` — still no source, no tests, no config.

- **`ARCHITECTURE.md`'s pipeline has no separate "parse" stage, even though `PROJECT_BRIEF.md` section 1 describes the pipeline as "capture → parse → classify → output."** Reason: parsing is inherent to each classifier's `classify()` — splitting it into its own stage would mean inventing a shared intermediate representation across 7 very different raw-text formats (tsc diagnostics, npm error codes, shell errors) for no real benefit, which conflicts with the "boring core, no cleverness" rule (section 4.3). Documented as a deliberate simplification in `ARCHITECTURE.md` section 3 rather than left unexplained.

- **The classifier authoring guide lives at `docs/CLASSIFIER_GUIDE.md`, not at the project root.** Reason: `PROJECT_BRIEF.md` section 5's required structure lists `ARCHITECTURE.md` and `README.md` explicitly at the root, but doesn't name a location for "a classifier authoring guide" (section 6 item 4) — and does list a `/docs` folder for "generated + hand-written docs." Read as the intended home for it.

- **README's before/after example, quickstart snippet, and "Supported error types" table were all verified against real, running code before being written down** — not just transcribed from the spec docs. Built the package, ran `npm pack`, installed the tarball into a genuinely fresh directory, and ran the exact quickstart code against a real failing command (`node -e "require('not-installed')"`), confirming the documented output shape. Also compiled a small external TypeScript consumer against the published `.d.ts` files to confirm the discriminated-union narrowing claim ("full type safety, no casting") actually holds outside this repo, not just in this repo's own test suite.

- **CRITICAL, found during Phase 6's final review: `install_failure` now matches BOTH `npm ERR!` (older npm, and the literal text of ERROR_SCHEMA.md's own section 6 example) and `npm error` (lowercase, no "!" — what npm 9+ actually emits, confirmed against a real `npm install <nonexistent-package>` using the npm 10.9.2 bundled with this project's own Node install).** Reason: the code-reviewer independently re-verified this classifier against real npm subprocess output (not just the hand-built `npm ERR!` fixtures every one of its own tests used) and found it silently produced `unknown_error` instead of `install_failure` against ANY current npm version — a functional regression against one of the 8 documented v1 error types, on the exact runtime (`"engines": {"node": ">=18"}`) this package targets. This is the same class of "assumed format vs. real format" bug that hit the tsc classifiers in Phase 5, except this one had NOT yet been smoke-tested against a real npm failure before this review — an oversight in Phase 5's own process, caught one phase late. Fixed with a prefix alternation (`npm (?:ERR!|error)`) and new regression tests built from real npm 10.9.2 output for both `E404` and `ERESOLVE`.

- **Fixed a broken README link:** `[PROJECT_BRIEF.md](PROJECT_BRIEF.md)` pointed at a filename that doesn't exist — the actual file (which must stay exactly where it is) is `PROJECT BRIEF.md`, with a space. Now links to `PROJECT%20BRIEF.md`.
